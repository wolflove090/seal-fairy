# 妖精コレクション機能要件書

## 背景と目的
- 現状の妖精出現処理は、シールごとに `hasFairy` の真偽値だけを持ち、妖精の種類を区別していない。
- 妖精ボタンや将来のコレクション画面追加を見据え、複数の妖精をデータとして登録し、シール配置時にその中からランダムで妖精を割り当てられる構成へ変更する必要がある。
- 妖精を見つけた実績をゲーム内セッション中は保持し、新規に見つけた妖精かどうかをログで判別できるようにする。
- 今回は永続保存を必須にしないが、将来的に `PlayerPrefs` 等へ保存しやすい責務分離を前提にする。

## スコープ内
- 複数妖精を登録できるデータ構造の定義
- シール配置時に妖精データからランダム選択して割り当てる仕様定義
- 配置済みシールがどの妖精を保持しているかを追跡する仕様定義
- セッション中の妖精獲得状態管理仕様の定義
- 新規発見か既発見かをログから判別できる仕様定義
- 将来の永続保存へ差し替えやすい保存責務の分離方針

## スコープ外
- `PlayerPrefs` や外部保存先への永続保存の実装
- 妖精図鑑 UI、詳細画面、一覧画面の実装
- 妖精ごとのレアリティ、出現率補正、解放条件の実装
- 妖精発見時の演出追加やサウンド追加
- 既存のシール選択 UI の見た目変更
- セッションをまたいだ獲得率分析や統計表示

## 機能要件

### 1. 妖精マスターデータ
- 妖精は複数件を登録できるデータとして管理する。
- 各妖精データには少なくとも以下の識別情報を持たせる。
- 一意な ID
- 表示名
- 出現抽選用の重み
- 必要に応じて将来 UI 表示へ使える参照情報
- 妖精データは `MonoBehaviour` の配列保持で管理し、Inspector から登録・差し替えできる構成にする。
- 実行時ロジックは妖精を配列添字ではなく ID で識別する。

### 2. シール配置時の妖精割り当て
- シールを新規配置したタイミングで、登録済み妖精データの中から 1 体を重み付きランダムで選択し、そのシールに割り当てる。
- 既存の `hasFairy` のような真偽値管理ではなく、「どの妖精が割り当てられたか」を保持する。
- ランダム対象は「現在登録されている妖精データ全件」とする。
- 妖精データが 0 件のときは、シール配置処理は継続できるが、妖精なしとして扱う。
- 妖精データが 0 件のときは、配置時に妖精関連ログを出さない。
- 同一妖精が複数枚のシールに割り当てられてもよい。

### 3. 配置済みシールと妖精情報の紐付け
- 配置済みシールごとに、未発見の妖精割り当て情報を保持する。
- シールを剥がして妖精判定を消費した後は、そのシールの未消費割り当て情報を取り除く。
- 既存の `StickerRuntimeRegistry` が担う「配置済みシール単位のランタイム追跡」を維持しつつ、保持内容を真偽値から妖精識別情報へ拡張できる構成にする。
- シール破棄時にランタイム情報が残留しないこと。

### 4. 妖精獲得状態の管理
- 妖精を発見した時点で、その妖精を「獲得済み」としてセッション中の状態へ反映する。
- 獲得状態はアプリ再起動で失われてもよい。
- 同一セッション内では、同じ妖精を再度見つけても獲得済み状態を維持する。
- 獲得状態の管理責務は、配置済みシール管理とは分離する。
- 将来 `PlayerPrefs` 等へ保存する際に、獲得済み妖精 ID 群を保存対象として取り出せる構成にする。

### 5. 発見ログ
- 妖精を発見した際は、少なくとも以下の 2 パターンをログから判別できるようにする。
- 初めて見つけた妖精
- 既に獲得済みの妖精を再度見つけた場合
- ログには妖精の表示名が含まれること。
- 新規発見ログは、後から Console を見返した際に新規獲得であることが明確に分かる表現にする。
- 既発見ログも、重複獲得であることが明確に分かる表現にする。
- 新規発見ログと既発見ログの最終文言は実装時に調整してよいが、少なくとも新規/既発見の区別と妖精名は必須とする。

### 6. 将来の永続保存を見据えた構造
- 今回の実装では永続保存を行わないが、獲得状態を扱う入口は 1 箇所に集約する。
- ゲームロジック側は「現在の獲得状態の読み書き」を抽象化された窓口経由で行い、保存方式の詳細を知らない構成にする。
- 初期実装ではメモリ上の状態管理を行い、将来的に `PlayerPrefs` 実装へ差し替えても、シール配置処理や剥がし処理の変更量が最小になること。

## 非機能要件
- 既存のシール配置フェーズとシール剥がしフェーズの挙動を壊さない。
- 妖精データ未登録時も NullReference や例外で停止しない。
- 新規クラス追加時は、妖精マスターデータ、配置済みシール割り当て、獲得状態管理の責務を混在させない。
- 既存ログと見分けやすい文面にし、デバッグ時に新規発見の有無を即座に判断できること。
- 将来 UI を追加する際に流用できるよう、表示名や ID を取得しやすい API 形状にする。
- 妖精発見時の演出は既存の `KiraKiraEffect` を共通利用し、妖精種類ごとの演出差し替えは行わない。

## 受け入れ条件
- Inspector 等で複数の妖精データを登録できる。
- 各妖精データに重みを設定できる。
- シールを複数回配置すると、各シールに対して登録済み妖精のいずれか 1 体が重み付きランダムで割り当てられる。
- シールを剥がしたとき、そのシールに割り当てられていた妖精を取得判定できる。
- 同じ妖精を初めて見つけた場合と 2 回目以降に見つけた場合で、Console ログの文面が区別できる。
- ログには対象妖精の名前が含まれる。
- アプリ起動中は、一度見つけた妖精が獲得済みとして扱われ続ける。
- アプリを再起動した後、獲得状態が初期化されても今回の要件では不具合としない。
- 妖精データが 0 件でもシール配置と剥がしは継続でき、妖精関連ログは出力されない。
- 将来保存方式を追加する際に、獲得状態管理の差し替え先が明確に存在する。
