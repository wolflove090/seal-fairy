# HUD操作ボタン白ベースブラッシュアップ要件書

## 背景と目的
- 現在の `HudScreen` にある `シールめくりへ`、`ショップ`、`妖精` の3ボタンは、単色グレーの矩形ボタンで実装されており、同画面内の所持金パネルやショップ画面内の白ベース装飾ボタンと比べて情報密度と質感が弱い。
- 既存 HUD はピンク基調のポップな見た目へ寄せているため、右側操作ボタンも同じ世界観に揃え、画面全体の統一感を上げたい。
- 今回は既存のボタン機能とレイアウト責務を維持したまま、3ボタンを「白ベースで、押しやすく、他UIと調和する操作パーツ」へ刷新する。

## スコープ内
- `HudScreen` 上の `ready-button`、`shop-button`、`fairy-button` の見た目再設計
- 白ベースのボタンプレート、枠、影、文字、補助装飾の仕様定義
- `シールめくりへ` ボタンと `ショップ` / `妖精` ボタンの共通トーン整理と差別化ルール定義
- hover / active 時の視覚フィードバック仕様定義
- UI Toolkit の UXML / USS を前提とした実装方針に繋がる命名・構造方針整理

## スコープ外
- `HudScreenBinder` が持つクリック処理、フェーズ遷移、ショップ起動、妖精画面起動ロジックの変更
- ボタン押下時のサウンド、アニメーション、パーティクル追加
- 所持金パネル、シール一覧パネル、ショップオーバーレイ、妖精コレクション画面自体のデザイン刷新
- ボタンの文言変更
- 解像度ごとの完全なレスポンシブ再設計

## 機能要件

### 1. 対象ボタン
- 対象は `ready-button`、`shop-button`、`fairy-button` の3要素とする。
- 各ボタンの既存名称とクリックイベント接続は維持する。
- 既存の表示位置は大きく変えず、右上のフェーズ変更ボタン、右下の `ショップ` / `妖精` ボタンという役割配置を維持する。

### 2. デザインの方向性
- 3ボタンともベースカラーは白系とする。
- 単なる白塗りではなく、既存ショップUIの閉じるボタンや価格プレートに近い「立体感のある白プレート」表現を採用する。
- 文字色は可読性を最優先しつつ、HUD全体のピンク基調と調和する濃色を使う。
- 枠線、内側余白、角丸、押し込み表現を用いて、タップ可能な要素であることが一目で分かる見た目にする。

### 3. ボタンごとの見せ分け
- `ready-button` はフェーズ変更の主操作であることが分かるよう、右下の2ボタンより視線誘導が強い見た目にする。
- `shop-button` と `fairy-button` は同一シリーズのサブ操作ボタンとして統一する。
- 3ボタンのシリーズ感は保ちつつ、`ready-button` はサイズ、文字強度、装飾のいずれかで優先度の差を表現する。
- 白ベース方針は共通化し、ボタンごとに配色ルールが分裂しないこと。

### 4. 装飾ルール
- ボタン外周には白プレート感を支える薄い境界色と下側重心のある縁表現を持たせる。
- 背景面には必要に応じて淡いピンクまたは暖色系の差し色をアクセントとして使ってよいが、主色は白であること。
- アイコンを新規追加しなくても成立する見た目を優先する。
- 装飾を増やしても文言可読性と押下領域を損なわないこと。

### 5. 状態変化
- hover 時は通常状態よりわずかに明るさまたは面色を変え、マウス操作時に反応が分かるようにする。
- active 時は既存ショップUIのボタンと同様に軽い沈み込み表現を持たせる。
- 状態変化は `background-color`、`translate`、`scale`、枠厚のいずれかで表現し、過剰な演出にしない。
- disabled 状態は現行仕様に存在しないため、新たな無効状態ロジックは追加しない。

### 6. レイアウト整合
- 既存の `top-bar` と `bottom-right-menu` の配置構造は維持し、他要素との重なりやはみ出しを発生させない。
- ボタンのサイズ調整を行う場合でも、1920x1080 基準で既存 HUD 内に自然に収まること。
- `ショップ` と `妖精` は縦積み配置を維持する。
- 長い文言ではないため、1行表示を維持し、文字切れや極端な縮小を発生させない。

### 7. 実装構造
- 実装は既存どおり UI Toolkit を使用する。
- 共通スタイルは USS 上でまとめ、3ボタンに対する重複定義を減らす。
- 必要であれば UXML に装飾用の子要素やラベルラッパーを追加してよい。
- UXML 構造を変更した場合でも、`HudScreenBinder` が参照するボタン要素の name は維持し、既存コード改修を最小限に抑える。

## 非機能要件
- 既存のショップ起動、妖精コレクション起動、フェーズ変更動作を壊さない。
- USS は役割が分かるクラス名・セレクタに整理し、今後ボタン意匠を再調整しやすい構成にする。
- 視認性を優先し、背景が明るいシーンでもボタン輪郭と文字が埋もれにくいこと。
- 既存の UI パフォーマンスを大きく悪化させる複雑な構造にしない。
- 16:9 前後の画面サイズで著しいレイアウト崩れを起こさないこと。

## 受け入れ条件
- `シールめくりへ`、`ショップ`、`妖精` の3ボタンが白ベースのデザインに変更されている。
- 3ボタンが現行 HUD 内で他UIと調和した見た目になっている。
- `ready-button` が補助ボタン2つより主操作として認識しやすい。
- `ショップ` / `妖精` の2ボタンが同一シリーズの見た目で揃っている。
- hover / active 時に視覚的フィードバックがある。
- クリック操作時に既存機能がそのまま動作する。
- ショップ画面表示、妖精コレクション表示、フェーズ切り替えの既存挙動に回帰がない。

## 未確定事項・確認事項
- 現時点ではなし。白ベースを前提に、差し色や装飾密度は既存 UI トーンに合わせて設計する。
