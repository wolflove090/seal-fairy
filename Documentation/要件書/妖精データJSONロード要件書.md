# 妖精データJSONロード要件書

## 背景と目的
- 現在の妖精マスタは `FairyCatalogSource` の Inspector 配列に直接保持されており、データ更新のたびにシーン上の設定変更が必要になっている。
- 妖精データを JSON へ分離し、ゲーム開始時に読み込む構成へ変更することで、妖精マスタの編集性と差分管理性を改善したい。
- 既存の妖精抽選、妖精コレクション表示、発見状態管理の利用側 API を大きく壊さず、データ供給方法のみを Inspector 依存から起動時ロードへ移行する。

## スコープ内
- 妖精マスタを JSON 形式で定義するためのデータ項目整理
- ゲーム開始時に妖精 JSON を読み込み、`FairyCatalogSource` から参照できる状態を作る仕様定義
- 既存利用箇所 (`TapStickerPlacer`、`HudScreenBinder` など) が JSON ロード後データを参照できる構成の要件整理
- JSON データ欠落時、ロード失敗時、必須項目不足時のフォールバック方針整理
- `Resources.Load<Sprite>` を使った画像参照ルール定義
- `Main.unity` に埋め込まれている既存妖精データの JSON への転記方針整理

## スコープ外
- 妖精抽選確率ロジック自体の変更
- 妖精発見状態の保存方式変更
- 妖精コレクション UI レイアウト変更
- Addressables 導入やアセット配信基盤の新規導入
- 妖精以外のシール、ショップ、通貨データの JSON 化

## 機能要件

### 1. 妖精マスタ JSON 定義
- 妖精マスタは複数件の妖精定義を保持できる JSON ファイルとして管理する。
- 各妖精定義は少なくとも `id`、`displayName`、`weight`、`iconResourcePath`、`favoriteStickerText`、`flavorText` を保持できる。
- `id` は妖精ごとに一意であり、既存の発見状態管理や抽選結果参照で継続利用できる値とする。
- `weight` は既存抽選ロジックが利用できる整数値とする。
- JSON は `Resources` 配下の Unity 実行時に読み取り可能な配置場所へ置く。
- `iconResourcePath` には `Resources.Load<Sprite>()` で読み込める相対パスを設定する。

### 2. 起動時ロード
- ゲーム開始時、妖精データ利用前に JSON を読み込み、妖精一覧をメモリ上に構築する。
- `FairyCatalogSource` は Inspector 配列ではなく、別ローダーまたはリポジトリが起動時に構築した妖精一覧を返せる。
- `TapStickerPlacer` の妖精抽選処理は、ロード完了後の妖精一覧を使って従来どおり抽選できる。
- `HudScreenBinder` の妖精一覧表示と詳細表示は、ロード完了後の妖精一覧を使って従来どおり描画できる。
- 現在 `Main.unity` に直列化されている 3 件の妖精データは、初期 JSON データへ転記する。

### 3. ロード失敗時の挙動
- JSON ファイルが見つからない、または JSON 解析に失敗した場合でも、ゲーム全体が例外で停止しない。
- JSON ファイル欠落または JSON 解析失敗時は `Debug.LogError` 相当のエラーログを出し、原因が追跡できる状態にする。
- ロード失敗時は妖精一覧を空として扱い、安全に継続できる。
- 不正な妖精定義 (`id` 欠落、`weight <= 0` など) は抽選対象外にできる。
- 一部レコード不正時は全件失敗にせず、不正レコードのみスキップする。
- 不正レコードをスキップした場合は、対象レコードを識別できる情報を含むエラーログまたは警告ログを出す。

### 4. 画像参照
- JSON には `Sprite` を直接埋め込まず、実行時に `Sprite` を解決できる参照情報を持たせる。
- 画像参照方式は `Resources.Load<Sprite>` に統一する。
- JSON には `Resources.Load<Sprite>` に渡すパスを記載する。
- 画像解決失敗時は、UI 表示側で null を安全に扱える状態を維持する。

### 5. 既存 API 互換
- `FairyCatalogSource.GetFairies()` の呼び出し側は、大きな責務変更なしで妖精一覧を取得できる。
- `FairyDefinition` の参照 API (`Id`、`DisplayName`、`Weight`、`Icon`、`FavoriteStickerText`、`FlavorText`) は、既存利用箇所の改修範囲を最小化できる形を維持する。
- 妖精一覧が未ロードまたは空の場合でも、既存 UI と抽選処理が null 参照で破綻しない。

## 非機能要件
- 1 回の起動時ロードで完結し、フレーム更新ごとに JSON を再読込しない。
- データロード責務、JSON 解析責務、`Sprite` 解決責務を分離し、責務境界を追いやすくする。
- JSON 追加・更新時にシーン YAML を直接編集しなくてよい運用にする。
- ロード失敗時に原因を追跡できるログを残す。
- 将来的な妖精データ追加時に、コード変更なしで JSON 追記のみを基本運用にできる構成を目指す。

## 受け入れ条件
- 妖精マスタが JSON ファイルで管理されている。
- ゲーム起動後、`FairyCatalogSource` 経由で JSON 由来の妖精一覧を取得できる。
- シール配置時の妖精抽選が、JSON から読み込まれた `weight` と `id` を用いて動作する。
- 妖精コレクション一覧と詳細画面が、JSON から読み込まれた表示名・画像・好きなシール・フレーバーを表示できる。
- JSON ファイル欠落または不正 JSON の場合でも、実行時例外でゲーム進行が停止しない。
- 画像参照に失敗した妖精がいても、一覧表示・詳細表示・抽選処理が致命的エラーにならない。

## 未確定事項・確認事項
- 初期 JSON データへ転記する既存妖精は、現在 `Main.unity` に登録済みの 3 件 (`バラちゃん`、`ウルフちゃん`、`もっさん`) を対象とする。
- JSON ロード失敗時にゲーム起動を止める必要はなく、エラーログを出したうえで安全継続する方針とする。
